# MVP 產品規格

> 本文件是 Shared Ledger 產品範圍、功能完成度、驗收條件與優先順序的唯一來源。技術實作請參考[架構說明](ARCHITECTURE.md)，上架文案請參考 [App Store 與 TestFlight](AppStore_TestFlight_zh-TW.md)。

## 產品目標

讓家庭、伴侶、室友與旅行群組能在同一個共享空間建立多個用途獨立的帳本，清楚知道每筆款項所屬帳本、來源帳戶、分類、分攤方式與最終結算結果。

## 核心使用流程

1. 建立群組並邀請成員。
2. 在群組內建立或選擇帳本，例如家庭日常、旅行或裝潢。
3. 在群組建立共用記帳帳戶，並在目前帳本建立階層式分類。
4. 使用群組帳戶在目前帳本新增收入、支出或帳戶轉帳。
5. 指定付款人、參與成員與分攤方式。
6. 查看跨帳本的群組帳戶餘額，以及目前帳本的成員淨額與月報表。
7. 依「誰應付給誰」的建議完成全額或部分結算。
8. 離線時持續記帳，恢復連線後同步共享資料。

## P0：完整 MVP 必須補齊

下列功能直接影響帳務正確性、共享可信度或核心流程，優先於 OCR、Widget 等增強功能：

1. **交易完整生命週期**：交易詳情、編輯、作廢／刪除確認與變更稽核；已參與結算的交易不得靜默改寫。
2. **完整分攤與多人付款**：平均、比例、指定金額、部分成員與多人付款；保存使用者選擇的分攤模式及輸入值，付款總額和分攤總額都必須等於交易金額，並依貨幣最小單位明確處理尾差。
3. **多帳本生命週期與資料隔離**：一個群組可建立多個帳本，並支援建立、重新命名、排序、切換、設定預設與封存；帳戶屬於群組並可供同群組的所有帳本共用，分類與交易各自只屬於一個帳本。交易不得引用其他群組的帳戶或其他帳本的分類。
4. **真實帳戶餘額與對帳**：在群組範圍內處理期初餘額、所有帳本交易與帳戶直屬調整對餘額的正確影響、帳戶明細、對帳日期，以及有歷史時只能封存不可刪除。
5. **債務與結算引擎**：逐成員淨額、「誰應付給誰」與最少筆數建議；支援全額／部分結算、外部付款備註、結算歷史與撤銷。
6. **搜尋、篩選與核對**：依關鍵字、日期、金額、帳戶、分類、付款人、參與成員與交易類型篩選，並提供月份分組與無結果狀態。
7. **群組與成員生命週期**：邀請重送與撤回、移除／退出、擁有權轉移；成員離開後保留顯示名稱快照，不破壞歷史帳務。
8. **角色權限真正生效**：擁有者、管理員、成員、唯讀成員的檢視、記帳、帳本管理、設定、邀請、結算與刪除權限必須一致生效。
9. **可用的總覽與月報表**：本月收入／支出、分類占比、待結算金額、成員淨額與月份切換使用目前帳本的真實資料；帳戶餘額使用群組內所有帳本交易與帳戶調整，畫面必須明確標示範圍。
10. **同步狀態與錯誤恢復**：呈現未登入、同步中、成功、離線、失敗與重試狀態，並完成雙帳號、離線恢復與衝突測試。
11. **資料可攜與刪除**：支援完整 CSV 匯出與系統分享；可刪除本機群組／個人資料，並清楚說明共享資料影響。
12. **通知與提醒最低版本**：群組邀請、交易修改與待結算通知可個別關閉；未授權通知時仍能正常使用 App。
13. **品質與可及性門檻**：完成核心計算、資料整合與關鍵流程測試；驗證 Dynamic Type、VoiceOver、深色模式、減少動態效果、金額格式與正體中文介面。

### P0-2 分攤與多人付款待完成明細

- 新增穩定的分攤模式：`equal`、`percentage`、`fixedAmount`；既有交易缺少模式時視為 `equal`，不得改變目前保存的 `EntrySplit.amount`。
- `EntrySplit.amount` 保存每位成員最後實際負擔金額；另保存比例或指定金額的原始輸入快照，讓交易詳情、編輯與稽核能還原使用者當時的設定。
- 平均分攤允許只選部分成員；比例分攤要求每位比例大於零且合計精確等於 100%；指定金額要求每筆非負且合計等於交易金額。
- 將目前單一 `LedgerEntry.payer` 擴充為付款明細，支援一位或多位付款人及各自付款金額；付款總額必須等於交易金額。
- 尾差依交易貨幣的最小單位計算，不得固定假設兩位小數；採可重現的分配順序，最終付款與分攤總額都必須精確等於交易金額。
- 新增與編輯使用同一套 Domain／repository 驗證；無效比例、無效指定金額、重複付款人、跨群組成員及封存成員都必須被資料層拒絕。
- 交易修改或作廢後重新計算成員淨額與結算結果，並保存修改前後的付款及分攤快照，不可靜默覆寫歷史。
- 單元測試至少涵蓋平均、比例、指定金額、部分成員、多人付款、零位與三位小數貨幣、正負尾差、編輯及無效輸入。

## MVP 驗收條件

### 群組、帳本、成員與權限

- 群組建立者自動成為擁有者。
- 每個群組建立時自動產生「主要帳本」，且任何時候至少保留一個啟用中的帳本與一個明確的預設帳本。
- 使用者可在同一群組建立及切換多個帳本；封存帳本不刪除既有帳務資料。
- 加入群組即可看見該群組全部帳戶與帳本；MVP 不提供逐帳本成員或單一本帳本分享。
- 擁有者、管理員、成員與唯讀四種角色的權限符合 P0 定義。
- 已撤回或失效的邀請不可加入；移除或離開的成員不破壞歷史帳務。
- 使用兩個不同 Apple Account 完成邀請、接受、編輯與同步驗證。

### 帳戶、分類、交易與結算

- 帳戶明確歸屬群組，並可被同群組的多個帳本共用；分類與交易明確歸屬單一帳本。跨群組帳戶或跨帳本分類關聯會被資料層拒絕。
- 帳戶餘額能由期初餘額、群組內所有帳本的相關交易與帳戶直屬調整正確推導；同群組帳戶間的轉帳不改變群組總資產，且不列入收入／支出。
- 一般交易一定屬於單一帳本；餘額調整直接屬於帳戶並保留原因與時間，不以無帳本交易表示。
- 分類在單一帳本內形成階層；已被交易使用的分類只能封存。
- 收入、支出與同帳本轉帳都能新增、核對、編輯或作廢；平均、比例與指定金額分攤皆可選部分成員並在詳情完整還原原始設定。
- 單人或多人付款的付款明細合計、分攤明細合計與交易金額三者一致；任何不一致都無法保存。
- 系統能正確產生成員淨額及結算建議，並保存部分／完整結算歷史。
- 金額計算精確，任何付款、分攤或結算都不產生無法解釋的尾差。

| 畫面或計算 | 預設資料範圍 |
| --- | --- |
| 帳戶餘額與帳戶明細 | 整個群組，包含所有帳本交易與帳戶調整；明細標示交易所屬帳本 |
| 收入、支出與分類報表 | 目前帳本；轉帳與餘額調整不列入收入／支出 |
| 成員淨額與結算 | 目前帳本；群組總覽只能彙總並清楚標示各帳本結果 |
| 交易搜尋與匯出 | 預設目前帳本；若提供跨帳本模式，必須明確標示範圍 |

### 報表、資料與同步

- 總覽、報表、搜尋與匯出均來自真實資料，結果能回到原始交易核對。
- 無網路時仍可新增與瀏覽本機資料，恢復網路後能同步。
- 未登入 iCloud、同步失敗與資料衝突都有清楚且可恢復的使用者體驗。
- V1／V2 舊資料升級後，帳戶保留在原群組並可供各帳本共用，分類與交易歸入原帳本或所屬群組的「主要帳本」，舊版餘額調整轉成帳戶直屬紀錄；餘額及歷史關聯保持一致，重複執行修復不產生重複資料。
- 刪除、匯出與共享資料影響有明確說明及確認步驟。

## P1：MVP 穩定後的高價值功能

- 重複交易、到期提醒與單次例外處理。
- 群組／分類預算、超支提醒與月結轉規則。
- 收據照片、附件、商家、標籤與保固備註。
- 多幣別、基準幣別、匯率日期、手動覆寫與結算匯率快照。
- 期間比較、趨勢、成員與分類分析，以及可回溯原始交易的洞察。
- 版本化 CSV 匯入、預覽驗證、重複偵測與失敗回滾。
- Face ID／Touch ID App Lock、背景遮罩與敏感通知選項。
- 常用交易範本、複製上一筆、內建計算機與最近選項。
- 每本帳本可釘選或隱藏常用群組帳戶；這只影響選單，不改變帳戶所有權與歷史資料。
- **多語系與在地化**：首波支援正體中文與英文，完整處理幣別、日期、時區、數字、複數與無障礙文案；詳細範圍與驗收條件如下。

### 多語系與在地化

- 首波支援正體中文（`zh-Hant`）與英文（`en`），預設依 iOS 系統語言顯示；架構需允許後續新增語言。
- 使用 Xcode String Catalog（`.xcstrings`）集中管理所有使用者可見文字，包括 SwiftUI 畫面、錯誤訊息、通知與 VoiceOver 標籤，不在程式碼中硬編碼顯示字串。
- 日期、時間、數字、百分比與金額依目前 `Locale` 顯示；貨幣格式以帳本、帳戶或交易保存的貨幣代碼為準，不只依裝置地區推測。
- 複數、語序與帶參數訊息使用 String Catalog 的變化規則，不以字串串接組成句子。
- 使用者自訂的群組、帳本、帳戶與分類名稱保留原文；App 內建的預設分類使用穩定識別碼對應各語言名稱，切換語言不得改變資料關聯。
- 新增功能時，同一個 PR 必須補齊正體中文與英文文案，並檢查長字串、Dynamic Type、VoiceOver 與不同螢幕尺寸的版面。
- 驗收時須確認兩種語言的核心流程沒有未翻譯 key，日期、金額及複數格式正確，切換語言後既有帳務資料保持不變。

## P2：差異化與平台整合

- OCR 收據辨識與使用者確認流程。
- Widget、App Intents／Shortcuts、Spotlight 與 Siri 快速記帳。
- Apple Watch 快速新增與查看待結算。
- 智慧分類、重複交易偵測與可解釋、可取消的自然語言建議。
- 付款請求或第三方支付深連結；MVP 只記錄結算，不直接保管或代收款項。

## P3：Apple Intelligence 與進階智慧功能

P3 建立在 P2 的 App Intents、Spotlight、OCR 與完整帳務規則之上。實作時以 Apple 當期的 [Foundation Models](https://developer.apple.com/documentation/foundationmodels) 與 [App Intents](https://developer.apple.com/documentation/appintents) 官方介面為準，不提前提高目前 iOS 17 核心功能的最低需求。

### 待加入功能

- 自然語言建立交易草稿，例如從「昨天晚餐 1,200 元，小明先付，我和小美各一半」產生日期、金額、帳戶、分類、付款與分攤候選值。
- 依目前群組與帳本資料提出分類、帳戶、分攤成員與重複交易建議，並解釋建議依據。
- 以自然語言查詢帳務，例如「這個月餐飲比上個月多多少」，答案可回到實際交易與報表核對。
- 產生期間摘要、異常支出提示與可執行的整理建議；不得把推測描述成已確認事實。
- 透過 App Intents／Siri AI 查詢帳本或建立草稿；正式儲存、結算、刪除、邀請與共享仍遵守 App 內相同權限及確認流程。

### 安全、隱私與帳務規則

- Apple Intelligence 只提供理解、建議與草稿；金額、分攤、餘額、債務、結算、權限及 migration 一律由可測試的 Domain／repository 規則決定，模型輸出不是帳務真實來源。
- 模型不得直接新增、修改、作廢或刪除帳務資料。所有會改變資料的結果先以結構化預覽呈現，經使用者確認並通過 repository 驗證後才可寫入。
- 使用 Foundation Models 的 guided generation／結構化輸出限制欄位型別與選項；模型產生的 UUID、金額、日期、帳戶、分類、成員及比例仍須重新查找並驗證，不可信任純文字輸出。
- 只提供完成當次請求所需的最小資料，且不得超出目前使用者可存取的群組與帳本 scope；角色權限、封存狀態及跨群組限制在 App 端強制執行，不能交由 prompt 約束。
- 群組名稱、交易備註、OCR 文字與附件內容視為不可信資料，不得被當成系統指令或繞過工具／repository 權限。
- 優先使用 Apple 系統提供的模型能力。若未來啟用 Private Cloud Compute、其他模型 provider 或第三方服務，必須在傳送前清楚揭露資料範圍、取得同意，並重新檢查隱私政策、App Privacy 與共享成員影響。
- 不以群組帳務、提示內容或模型回覆訓練自有或第三方模型；除非使用者主動提交診斷，不保存完整 prompt、回覆或 Foundation Models session transcript。
- AI 產生的草稿與洞察需標示為建議，顯示所用帳本、日期範圍與來源交易；無法由真實資料核對的數字不得顯示為確定金額。

### 可用性、降級與驗收

- 每次使用前檢查系統模型的 runtime availability，涵蓋不支援的裝置、Apple Intelligence 未啟用、模型尚未就緒及語言不可用；不可只以 OS 版本或機型推測。
- Apple Intelligence 不可用、逾時或生成失敗時，核心記帳、搜尋、報表、匯出與結算仍完整可用，並提供手動輸入或既有規則式功能，不出現無法離開的 loading 狀態。
- App Intents 與模型工具只能呼叫正式 Domain／repository API，不建立第二套帳務邏輯；需要確認的操作在 Siri、Shortcuts 與 App 內維持一致。
- 評估至少涵蓋正體中文與英文、不同日期／幣別格式、比例與指定金額分攤、模糊敘述、惡意備註、跨群組請求、封存資料、無模型與中斷恢復。
- 數字正確性以固定資料集驗證：模型可以選擇查詢或摘要方式，但所有總額、差額、比例與結算結果必須由 deterministic calculator 產生並能回溯來源。
- P3 功能完成需包含真實支援裝置驗證、Dynamic Type、VoiceOver、錯誤與降級狀態、效能／耗電觀察，以及使用者可取消生成與清除 AI 暫存資料。

## 功能完成定義

功能只有在資料模型與遷移、正確的群組／帳本 scope、權限、離線與同步、錯誤狀態、可及性、測試及真實資料畫面都完成後，才可標記完成。只有畫面入口、靜態卡片、資料欄位或 happy path 不算完成。
